# AIEcome 公海智能比价采购平台

面向 B2B/B2C 通用采购场景的产品需求展示页。它不是普通商城，而是一个把 **多商家货源公海化、采购需求 AI 结构化、非标规格自动换算、质量优先智能比价、跨商家订单分账** 串成闭环的平台方案。

> 产品原始需求来自 `电商平台.docx`。本页面已将文档内容整理成可浏览、可演示、可评审的 GitHub Pages 站点。

## 这个项目解决什么问题

传统电商更适合“用户知道商品名，直接搜索购买”。本项目面向更复杂的采购场景：

- 买家常用口语、图片、OCR、长文本描述需求，规格不统一。
- 同款商品可能来自多个商家，价格、税费、运费、交期、质检等级都不同。
- 采购方真正关心的不是裸价最低，而是质量可靠、综合成本低、交付风险小。
- 平台需要治理商品标准、行业词库、单位换算、虚假报价和低质供货。
- 跨商家下单后，需要一笔付款、多方履约、多方分账和可追溯对账。

因此页面围绕一个核心闭环组织：

```text
商家商品入驻
→ 标准化归入公海商品池
→ 买家 AI 语音/OCR/文本提交需求
→ 结构化解析采购参数
→ 统一规格换算和含税含运费比价
→ 质量 60% + 价格 40% 综合排序
→ 跨商家下单、支付、分账、售后
→ 评价和履约数据回流质量评分
```

## 核心能力

| 能力 | 用户价值 | 开发重点 |
|------|----------|----------|
| 公海商品池 | 所有商家同款货源可统一检索、同屏比价 | SPU/SKU、商家报价、库存、质检、标准类目 |
| AI 采购入口 | 买家不用精确搜索词，也能发起采购 | ASR/OCR/长文本 → LLM 结构化 JSON |
| 规格标准化 | 解决吨/公斤、箱/件、型号别名等非标问题 | 单位换算、包装解析、行业规则库 |
| 质量优先排序 | 避免只按低价推荐劣质货源 | 质检资质、历史评价、商家资质、售后履约 |
| 跨商家交易 | 一次付款、多方履约、多方分账 | 子订单拆分、担保交易、佣金、提现、对账 |
| 平台治理后台 | 公海池长期可运营、可审计、可调参 | 权重配置、词库维护、异常预警、资金风控 |

## 页面内容

- **项目定位**：说明为什么它不是普通商城，而是 1688 + 京东企业购 + AI 采购助手 + 价格比较网站的综合体。
- **高保真原型**：买家采购端、商家供货端、平台总后台三端均可独立打开，覆盖核心表单、弹窗、排序、审核、结算和状态反馈。
- **用户需求**：按采购买家、入驻商家、平台运营、资金/第三方服务方拆解痛点与功能响应。
- **交互演示**：公海商品池、规格换算、AI 采购助手、多维比价引擎。
- **核心难点**：商品标准化、规格换算、商品匹配、质量评分。
- **数据模型与架构**：SPU/SKU、公海池、微服务拆分、AI 解析与比价服务。
- **三端功能**：买家端、商家端、平台管理后台。
- **落地路线**：从 MVP 公海池到 AI 解析、比价引擎、分账交易和运营治理。
- **验收标准**：按需求文档列出可验证交付项。

## 需求红线

开发时必须保留以下业务约束：

- 所有商家商品强制进入公海池，不做独立商家私有商品库。
- 比价必须统一规格、统一含税含运费综合成本，不能只比较裸价。
- 默认排序必须质量权重大于价格权重：质量 60%，价格 40%。
- AI 语音输入是买家核心入口，优先级高于传统文字搜索。
- 商家只能管理自有商品，不能干预公海标准、其他商家报价或排序算法。
- 跨商家订单必须支持子订单拆分、多方分账、售后退款和资金对账。
